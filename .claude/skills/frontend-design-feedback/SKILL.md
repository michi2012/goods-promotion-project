# 디자인 다듬기 루프 (Playwright MCP 기반 시각 피드백)

사용자는 프론트엔드 디자인 어휘에 익숙하지 않을 수 있다. "버튼이 너무 커요", "가운데로
옮겨주세요" 같은 평범한 말만으로도 반복 개선이 가능하도록, **작게 수정하고 바로 보여주는
루프**를 돈다. 한 번에 화면 전체를 새로 디자인하지 않는다.

## 1. 사전 조건

- `frontend/`에서 `npm run dev`가 구동 중이어야 한다 (`localhost:5173`). 떠 있지 않으면
  `run_in_background: true`로 실행한다 (CLAUDE.md 장기 실행 명령 처리 원칙).
- Playwright MCP(`mcp__playwright__*`)가 연결되어 있어야 한다. `.mcp.json` /
  `.claude/settings.json`의 `enabledMcpServers`에 `playwright`가 등록되어 있다.
- API 데이터까지 포함한 화면을 확인하려면 해당 백엔드(예: serverB)도 로컬에서 떠 있어야
  한다. 떠 있지 않으면 API 호출은 에러/Bad Gateway로 보이는데, **레이아웃만 다듬을
  때는 무시해도 된다** — 백엔드 의존 여부를 먼저 사용자에게 확인할 필요는 없다.

## 2. 루프 절차

1. **이동**: `mcp__playwright__browser_navigate`로 확인할 라우트로 이동한다
   (예: `http://localhost:5173/`). 이전 상태(입력값, 에러 등)가 남아있으면 같은 URL로
   다시 `browser_navigate`하면 새로고침되어 초기 상태로 돌아온다.
2. **스크린샷**: `mcp__playwright__browser_take_screenshot`(`fullPage: true`)로 전체
   화면을 캡처한다. "모바일에서는 어떻게 보여요?" 같은 화면 크기 관련 피드백이 나오면
   `mcp__playwright__browser_resize`로 뷰포트를 좁게(예: 375×667) 바꾼 뒤 다시
   캡처해 비교한다. 화면에 로딩/에러/데이터 표시 등 여러 상태가 있다면, 초기 상태만으로
   끝내지 않고 주요 상태도 한 번씩 재현해(예: 조회 버튼 클릭) 함께 캡처한다.
3. **설명**: 스크린샷을 보여주고, 화면 구성을 **평범한 말**로 설명한다 (전문 용어 대신
   "위쪽에 ~가 있고, 가운데에 ~카드가 있다" 식). 어떤 점이 비어 보이는지/어색한지도
   함께 짚어준다.
4. **피드백 수신**: 사용자가 평범한 말로 피드백을 준다 (예: "빈 공간을 다른 걸로
   채워줘", "버튼이 너무 커요"). 모호하면, 추측보다는 **가장 단순하고 되돌리기 쉬운
   해석**으로 한 번 시도해보고 결과로 보여준다 — 디자인 의도를 길게 묻지 않는다.
5. **코드 수정**: `frontend` skill의 컨벤션(shadcn/ui 컴포넌트, `className` 합성,
   기존 페이지 구조)을 따라 **작은 단위로** 수정한다.
6. **재확인**: 1~2번을 반복해 변경 결과를 다시 스크린샷으로 보여준다.
7. 사용자가 "이대로 하자" 등으로 만족을 표현하면 루프를 종료한다. 더 다듬을 점이
   있으면 4번부터 반복한다.

## 3. 주의사항

- 한 번의 루프에서 여러 군데를 동시에 바꾸지 않는다 — 무엇이 어떤 피드백 때문에
  바뀌었는지 추적 가능해야 한다.
- `frontend/src/components/ui/**`(shadcn 생성 컴포넌트)는 직접 고치지 않는다.
  `className` 합성이나 페이지/컴포넌트 레벨 래퍼로 조정한다 (`frontend` skill 3절).
- 콘솔 에러가 보이면(예: API 502) 디자인 피드백과 무관한지 먼저 구분한다 — 백엔드
  미구동으로 인한 에러라면 디자인 루프와 별개 사항으로 간단히 언급만 하고 넘어간다.

## 4. frontend / frontend-testing과의 관계

- `frontend`: 화면/컴포넌트를 **처음 구현**할 때의 컨벤션 (구조, API 연동, shadcn 사용법).
- `frontend-testing`: **자동화된** 검증 (Vitest, MSW, Playwright E2E — 시각 회귀/접근성
  포함, 회귀 방지 목적).
- `frontend-design-feedback`(이 문서): 구현 후 **사람이 보고 즉석에서 다듬는** 1회성
  피드백 루프. 자동화 테스트를 대체하지 않는다 — 디자인이 확정되면 필요 시
  `frontend-testing`의 시각 회귀 스냅샷을 갱신한다.

## 5. 자가 점검
- [ ] 변경 전/후 스크린샷을 모두 보여줬는가?
- [ ] 한 번에 하나의 피드백만 반영했는가?
- [ ] shadcn 생성 컴포넌트(`components/ui/**`)를 직접 고치지 않았는가?
- [ ] 사용자의 만족 표현("이대로 하자" 등)으로 루프를 종료했는가?
