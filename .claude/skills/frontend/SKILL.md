# 프론트엔드 작업 매뉴얼

## 0. 작업 시작 전 필수 확인 사항

1. **기존 컴포넌트 컨벤션 파악**
    - 비슷한 컴포넌트 1-2개를 먼저 읽는다.
    - 파일 구조 (한 파일 vs 폴더), 명명, props 전달 방식, 스타일링 방식을 그대로 따른다.

2. **디자인 시스템/UI 라이브러리 확인**
    - 프로젝트에 디자인 시스템이나 UI 라이브러리(shadcn, MUI, Antd 등)가 있는가?
    - 있다면 그 컴포넌트를 우선 사용. 직접 만들지 마라.

3. **상태 관리 패턴 확인**
    - 로컬 상태 / Context / 외부 라이브러리(Redux, Zustand, Jotai 등) 중 무엇을 쓰는가?
    - 새 상태가 정말 전역이어야 하는지 다시 한 번 생각한다. 가능하면 로컬에 둔다.

## 1. 보편 규칙

### 1-1. 컴포넌트 책임 분리
- 한 컴포넌트 = 한 책임. 너무 많은 일을 하는 컴포넌트는 분리한다.
- 데이터 페칭 / 비즈니스 로직 / 표현(presentation)을 적절히 분리한다.
- 그렇다고 단순한 컴포넌트를 과도하게 쪼개지 마라 (Simplicity First).

### 1-2. 상태 처리
모든 인터랙티브 컴포넌트는 다음 상태를 명시적으로 처리한다:
- 로딩 중
- 에러 발생
- 빈 데이터 (empty state)
- 정상 데이터

UX적으로 의미 있는 경우만. 단순한 로컬 버튼에 4가지 상태를 다 만들지 마라.

### 1-3. 폼 처리
- 프로젝트의 기존 폼 라이브러리(react-hook-form 등)를 그대로 사용.
- 클라이언트 검증과 서버 검증을 모두 한다. 클라이언트 검증만 믿지 마라.
- 제출 중 버튼 비활성화 + 시각적 피드백.
- 에러 메시지는 해당 필드 근처에 표시.

### 1-4. 접근성 (Accessibility)
- 모든 인터랙티브 요소(버튼, 링크, 인풋)는 키보드로 접근 가능해야 한다.
- 의미 있는 요소에 `aria-label` 또는 `aria-labelledby`.
- 이미지에 `alt` 텍스트. 장식용은 `alt=""`.
- 폼 인풋에 `<label>` 연결.
- 색상만으로 정보를 전달하지 마라.

### 1-5. 스타일링
- 기존 방식을 따른다 (Tailwind / CSS Modules / styled-components / SCSS 등).
- 인라인 스타일 사용 금지 (기존 코드가 일관되게 인라인을 쓰는 게 아니라면).
- 매직 넘버 금지 (`marginTop: 17px` 같은 것). 디자인 토큰 / 변수 사용.
- 반응형: 디자인이 명시되어 있지 않은 경우 사용자에게 확인.

### 1-6. 타입 (TypeScript)
- 프로젝트가 TypeScript를 사용 중이라면 `any` 사용 금지.
- props는 명시적 인터페이스/타입 정의.
- 외부 데이터(API 응답 등)는 타입 가드 또는 검증 라이브러리로 좁힌다.

### 1-7. 성능
- 불필요한 리렌더 주의. 단, 측정 없이 `useMemo`/`useCallback`을 무차별로 박지 마라.
- 큰 리스트는 가상화(react-window 등) 고려.
- 이미지는 적절한 포맷/크기/lazy loading 사용.
