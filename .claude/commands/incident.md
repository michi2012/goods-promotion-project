# /incident — 장애 RCA 작성

장애 발생 시 사실 수집 → 타임라인 정리 → RCA 문서 생성까지 가이드한다.

**사용법**
- `/incident` — 대화형으로 사실 수집 후 RCA 생성
- `/incident {제목}` — 제목을 미리 지정하고 시작

---

## Step 1. 사실 수집

사용자에게 아래 항목을 질문한다. 한 번에 전부 물어보지 말고,
**이미 제공된 정보는 건너뛰고** 빠진 것만 묻는다.

```
필수 수집 항목:
1. 장애 시작 시각 (최대한 정확하게, 예: 14:23)
2. 장애 종료 시각 또는 현재 진행 중 여부
3. 영향 범위 (어떤 서비스/API/사용자가 영향받았나)
4. 증상 (에러 메시지, 지표 이상, 사용자 신고 등)
5. 트리거 (배포, 트래픽 급증, 외부 의존성 장애 등 — 불명확하면 "미확인"도 가능)
6. 조치 내용 (재시작, 롤백, 설정 변경 등)
7. 근본 원인 (현재까지 파악된 것, 불명확하면 가설로)
```

수집이 완료되면 Step 2로 진행한다.

---

## Step 2. 관련 커밋 조회

장애 시점 전후 배포 여부를 확인하기 위해 아래를 실행한다.

```powershell
# 최근 10개 커밋 + 태그 확인
git log --oneline -10
git tag --sort=-creatordate | Select-Object -First 5
```

배포가 트리거로 의심되면 해당 커밋 diff를 추가로 확인한다.

---

## Step 3. RCA 문서 생성

수집한 사실을 바탕으로 아래 형식의 문서를 생성한다.
**추측은 명확히 "가설:"로 표기한다. 확인되지 않은 사실을 단정하지 않는다.**

```markdown
# Incident Report — {제목}

| 항목 | 내용 |
|------|------|
| 상태 | 🔴 진행중 / ✅ 해소 |
| 시작 | YYYY-MM-DD HH:MM (KST) |
| 종료 | YYYY-MM-DD HH:MM (KST) / 진행중 |
| 지속 시간 | N분 |
| 심각도 | P1 / P2 / P3 |
| 영향 범위 | {서비스명}, {영향받은 사용자/요청 수} |

## 증상
{관찰된 현상을 사실 기반으로 서술. 에러 메시지, 지표 수치 포함}

## 타임라인

| 시각 | 이벤트 |
|------|--------|
| HH:MM | {사실} |
| HH:MM | {조치} |
| HH:MM | {해소 확인} |

## 근본 원인
{확인된 원인}
가설: {아직 미확인된 부분}

## 조치 내용
- {실제 취한 조치}

## 재발 방지
| 액션 | 담당 | 기한 |
|------|------|------|
| {예방 조치} | {담당자} | {날짜} |

## 참고
- 관련 커밋: {hash}
- 관련 PR: #
- 모니터링 링크:
```

---

## Step 4. 파일 저장

생성된 문서를 저장할 경로를 제안한다.

```
docs/incidents/YYYY-MM-DD-{kebab-case-제목}.md
```

저장할까요? (y/n)

사용자가 승인하면 해당 경로에 파일을 생성한다.
`docs/incidents/` 디렉토리가 없으면 함께 생성한다.

---

## 심각도 기준 (참고)

| 등급 | 기준 |
|------|------|
| P1 | 전체 서비스 중단, 결제 불가, 데이터 유실 |
| P2 | 핵심 기능 일부 장애, 성능 심각 저하 |
| P3 | 일부 사용자 영향, 우회 가능 |
