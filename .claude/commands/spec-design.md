# /spec-design — 엔티티/API 계약 설계

정식 요구사항 명세서(`.md`)를 읽어, 새로 추가/변경되는 엔티티와 API 계약(엔드포인트, 요청/응답 필드, 상태 코드)을 설계 문서로 작성한다.
`/spec-to-tickets` 전에 실행하면, 명세서에서 쪼개진 여러 티켓이 동일한 계약을 참조할 수 있다.

**사용법**
- `/spec-design {명세서 경로}` — 해당 명세서를 읽어 설계한다
- `/spec-design` — 대화 중 붙여넣은 명세서 내용으로 설계한다

---

## Step 1. 명세서 입력 받기

파일 경로가 주어지면 해당 파일을 읽는다. 그렇지 않으면 사용자가 붙여넣은 내용을 명세서로 사용한다.
명세서가 없으면 사용자에게 요청한다.

`docs/system-design.md`가 존재하면 함께 읽는다 — 시스템 전체의 엔티티/서비스 경계 뼈대(`/system-design` 결과물)로 간주하고, 이번 설계가 그 경계와 일치하는지 Step 3에서 확인한다.

---

## Step 2. 변경 필요 여부 판단

명세서의 `## 시나리오 (Given-When-Then)`, `## 수용 기준`을 보고 판단한다.

- 기존 엔티티/API로 시나리오를 모두 처리할 수 있으면 → "엔티티/API 변경 없음 — 기존 구조로 충분합니다"라고 안내하고 종료한다. 이 경우 설계 문서를 생성하지 않으며, `/spec-to-tickets`로 바로 진행할 수 있음을 안내한다.
- 새 엔티티, 필드/관계 변경, 새 API 엔드포인트 중 하나라도 필요하면 → Step 3으로 진행한다.

---

## Step 3. 엔티티 변경안

새로 추가되거나 변경되는 엔티티만 작성한다. 변경 없는 기존 엔티티는 나열하지 않는다.
명세서에 없는 내용을 임의로 지어내지 않는다 — 추측이 필요한 부분은 "가정:"으로 명시한다.

```markdown
### {엔티티명} (신규 / 필드 추가 / 관계 변경)

| 필드 | 타입 | 제약 | 설명 |
|---|---|---|---|
| {필드명} | {타입} | {NOT NULL 등} | {설명} |

관계: {다른 엔티티와의 관계 — 있다면}
```

---

## Step 4. API 계약 초안

새로 추가되거나 변경되는 엔드포인트만 작성한다.

```markdown
### {METHOD} {경로}

{한 줄 설명}

**요청**
| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|

**응답**
| 필드 | 타입 | 설명 |
|---|---|---|

**상태 코드**
| 코드 | 상황 |
|---|---|
```

---

## Step 5. 설계 문서 생성 및 저장

Step 3, 4의 결과를 아래 형식으로 합친다.

```markdown
# 설계: {명세서 제목}

- 관련 명세서: {명세서 경로}

## 엔티티 변경
{Step 3 결과 — 변경 없으면 "엔티티 변경 없음"}

## API 계약
{Step 4 결과 — 변경 없으면 "API 변경 없음"}

가정: {불명확해서 임의로 채운 부분이 있다면 명시}
```

저장 경로를 제안한다.

```
docs/specs/{명세서와 동일한 kebab-case-제목}-design.md
```

저장할까요? (y/n)

사용자가 승인하면 해당 경로에 파일을 생성한다.
저장 후, 이어서 `/spec-to-tickets {명세서 경로}`로 티켓을 생성할 수 있음을 안내한다 — 같은 디렉토리의 `-design.md`는 `/spec-to-tickets`가 자동으로 함께 참조한다.
