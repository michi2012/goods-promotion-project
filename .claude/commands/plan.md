# /plan 명령어 — 작업 시작 전 표준 프로토콜

너는 지금부터 아래 6단계를 **엄격히 순서대로** 수행한다. 단계를 건너뛰거나, 코드를 미리 작성하거나, 단계를 합쳐서 실행하면 안 된다.

---

## Step 1. 가정 표면화 (Surface Assumptions)
사용자의 요청을 받자마자 다음을 명시적으로 나열한다. 절대 추측해서 진행하지 마라.

### 1-0. Linear 이슈 연동 확인 (선택)

사용자의 요청에 Linear 이슈 식별자(예: `MIC-5`)가 포함되어 있으면 `get_issue`로 해당 이슈를 조회한다.
- 이슈의 `title`·`description`을 "목표"·"성공 기준" 초안의 출발점으로 삼는다.
- 이슈에 이미 명시된 내용은 1-1~1-5에서 다시 묻지 않는다. 이슈에 없는 정보만 가정·질문 대상으로 삼는다.
- 이슈 description에 "관련 설계: {경로}"가 포함되어 있으면 해당 설계 문서(`docs/specs/*-design.md`, `/spec-design` 결과물)를 함께 읽는다. 문서에 정의된 엔티티/API 계약은 합의된 것으로 간주하고 1-1~1-5에서 다시 가정·질문하지 않는다.
- 이슈 description에 "## PR 전 체크리스트" 섹션이 있으면, 그 항목을 Step 4 `docs/checklist.md`의 "최종 검증"에 포함한다 (단, "성공 기준" 초안의 출발점에서는 이 섹션을 제외한다).
- 위 항목이 `docs/checklist.md` 템플릿의 기존 "최종 검증" 항목과 의미가 중복되면, 중복 추가하지 않고 기존 항목을 유지한다.
- 이슈 식별자가 없으면 이 단계를 건너뛴다.

### 1-1. 내가 가정하고 있는 것
사용자가 명시하지 않았지만 내가 임의로 해석한 모든 지점을 나열한다. 예를 들어 "로그인 기능 만들어줘"라면:
- 인증 방식 (JWT / 세션 / OAuth 등)
- 비밀번호 저장 방식 (bcrypt / argon2 등)
- 세션 만료 정책
- 비밀번호 정책 (최소 길이, 특수문자 등)
- 이메일 인증 필요 여부

### 1-2. 모호한 부분
하나의 요청이 여러 방식으로 해석될 수 있는 모든 지점.

### 1-3. 트레이드오프
선택지가 있다면 각 선택지의 장단점을 표로 정리.

| 선택지 | 장점 | 단점 | 권장 상황 |
|---|---|---|---|
| A | ... | ... | ... |
| B | ... | ... | ... |

### 1-4. 더 단순한 대안
사용자가 요청한 것보다 더 간단히 해결될 수 있는 경우, 그 대안을 제시한다. (Simplicity First 원칙)

### 1-5. 사용자에게 질문
위 내용을 정리한 후, 명확히 답변이 필요한 질문 목록을 만들어 사용자에게 묻는다.

**🛑 사용자가 답변하기 전에는 Step 2로 넘어가지 마라. 절대.**

## Step 2. `docs/plan.md` 작성
사용자 답변을 받은 후, 다음 형식으로 작성한다.
파일을 처음 작성할 때 내부 Write 도구에서 에러가 발생하지 않도록, 파일이 존재하지 않는다면 PowerShell 명령어(예: New-Item -Path docs\plan.md -ItemType File)를 사용해 빈 파일을 먼저 생성한 후 내용을 작성하라. 경로 작성 시 윈도우 환경에 맞게 백슬래시(\)를 사용하라.

```markdown
# 계획서: [작업 제목]

- 작성일: YYYY-MM-DD
- 관련 이슈/티켓: (있다면 — 1-0에서 Linear 이슈를 찾았다면 식별자와 URL을 채운다, 예: MIC-5 / https://linear.app/...)

## 목표
1-2문장으로 명확히. 모호한 표현 금지.

## 성공 기준
이 작업이 "완료"되었다고 판단할 객관적 기준 3-5개. 각 기준은 검증 가능해야 한다.
- [ ] 기준 1 (어떤 명령어/테스트로 확인하는지 명시)
- [ ] 기준 2
- [ ] 기준 3

## 비범위 (Out of Scope)
이번 작업에서 의도적으로 다루지 않는 것. 추후 작업으로 분리할 항목.
- ...
- ...

## 단계별 작업 계획 (최대 7단계)

### 단계 1: [제목]
- 변경 파일: src/..., tests/...
- 변경 내용 요약: 2-3문장
- 검증 방법: 정확한 명령어 또는 테스트 케이스
- 롤백 방법: 이 단계가 실패했을 때 되돌리는 방법
- 예상 소요: 짧음 / 보통 / 김

### 단계 2: [제목]
(동일 형식)
...

## 리스크 및 대응
- 리스크 1: ... → 대응: ...
- 리스크 2: ... → 대응: ...

## 의존성
다른 코드/데이터/외부 서비스 의존성이 있다면 명시.

```

## Step 3. `docs/context.md` 작성

미래의 나(또는 다른 개발자)가 이 결정을 이해할 수 있도록 작성한다.
파일을 처음 작성할 때 내부 Write 도구에서 에러가 발생하지 않도록, 파일이 존재하지 않는다면 PowerShell 명령어(예: New-Item -Path docs\context.md -ItemType File)를 사용해 빈 파일을 먼저 생성한 후 내용을 작성하라. 경로 작성 시 윈도우 환경에 맞게 백슬래시(\)를 사용하라.

```markdown
# 맥락 노트: [작업 제목]

## 왜 이 방식을 선택했는가
... (배경, 제약조건, 사용자와의 대화 결과)

## 검토했으나 채택하지 않은 대안
### 대안 A
- 무엇: ...
- 왜 안 썼나: ...

### 대안 B
- 무엇: ...
- 왜 안 썼나: ...

## 기존 코드베이스 컨벤션
- 디렉토리 구조: ...
- 명명 규칙: ...
- 에러 처리 패턴: 예시 파일 경로
- 테스트 구조: 예시 파일 경로

## 관련 파일/위치
- src/... — 어떤 역할
- src/... — 어떤 역할

## 외부 참조
문서 링크, RFC, 디자인 문서 등

```

## Step 4. `docs/checklist.md` 작성

작업 진행 추적용. 단순하고 실용적으로.
파일을 처음 작성할 때 내부 Write 도구에서 에러가 발생하지 않도록, 파일이 존재하지 않는다면 PowerShell 명령어(예: New-Item -Path docs\checklist.md -ItemType File)를 사용해 빈 파일을 먼저 생성한 후 내용을 작성하라. 경로 작성 시 윈도우 환경에 맞게 백슬래시(\)를 사용하라.

```markdown
# 체크리스트: [작업 제목]

- 마지막 업데이트: YYYY-MM-DD HH:MM

## 진행 상황
- [ ] 단계 1: ... 구현
  - [ ] 검증 통과 (<명령어>)
  - [ ] 코드리뷰 통과
- [ ] 단계 2: ... 구현
  - [ ] 검증 통과
  - [ ] 코드리뷰 통과
- [ ] 단계 N: ...

## 최종 검증
- [ ] 모든 단위 테스트 통과
- [ ] 모든 통합 테스트 통과
- [ ] 린터 통과
- [ ] 타입 체크 통과 (해당하는 경우)
- [ ] 변경 사항이 plan.md의 "비범위"를 침범하지 않았는지 확인
- [ ] 의도하지 않은 파일 변경이 없는지 git diff로 최종 확인

## 발견 사항 (작업 중 별도 처리 필요한 것)
- ...
- ...

```

## Step 5. 🛑 정지

위 세 파일을 출력하거나 요약 보고한다. 그리고 멈춘다.
사용자가 다음 중 하나로 명시적 승인을 하기 전에는 **단 한 줄의 코드도 작성하지 않는다**:

* "승인", "go", "진행", "approved", "OK"
* 또는 수정 지시 후의 재승인

사용자가 수정을 요청하면 Step 2~4를 다시 수정한 후 다시 정지한다.

### Linear 연동 후속 처리 (1-0에서 이슈를 찾은 경우에만)

상태 전환(`In Progress` 등)은 GitHub-Linear 자동 연동(브랜치/PR 생성 시 트리거)에 맡긴다 — 수동 전환은 자동화와 중복되므로 하지 않는다.

이 자동 연동이 동작하려면 브랜치명에 이슈 식별자가 포함되어야 한다 — `feature/mic-{n}-{설명}` 형식(예: `feature/mic-15-promotion-api`)으로 브랜치를 생성할 것을 사용자에게 안내한다.

승인을 받으면, Step 6(코드 작성)을 시작하기 전에 다음만 수행한다:
1. `save_comment`로 해당 이슈에 `docs/plan.md` 경로를 코멘트로 남긴다 (예: "계획서 작성 완료: docs\\plan.md").

## Step 6. 단계별 실행 (승인 후에만)

승인 후 다음 규칙을 엄격히 지킨다.

### 6-1. 절대 규칙

* **한 번에 한 단계만 실행한다.** 여러 단계를 동시에 진행하지 마라.
* 각 단계를 끝낸 직후 즉시 검증을 실행한다. 검증 명령어와 결과를 모두 보고한다.
* 검증 실패 시 즉시 정지하고 사용자에게 보고한다. 다음 단계로 가지 마라.
* 매 단계 완료 후 `docs/checklist.md`를 업데이트한다.

### 6-2. 단계 진행 중 새로운 사실이 발견되면

계획에 없던 변경이 필요해 보이면 즉시 정지하고 사용자에게 보고한다:

* 무엇을 발견했는지
* 왜 계획이 불충분한지
* 어떻게 수정하면 좋겠다는 제안

사용자 결정 전까지 진행하지 마라.

### 6-3. 단계 보고 형식

각 단계 완료 시 다음 형식으로 보고:

```markdown
# 단계 N 완료 보고

## 변경한 파일:
- path/to/file1.ts — 무엇을
- path/to/file2.ts — 무엇을

## 실행한 검증:
$ <정확한 명령어>
<출력 요약 또는 핵심 부분>

- 결과: ✅ 통과 / ❌ 실패
- 다음 단계: 단계 N+1로 진행할까요?

```

### 6-4. 최종 완료 시

모든 단계 완료 후 다음을 수행:

1. `docs/checklist.md`의 모든 항목 체크 확인
2. `git diff --stat` 또는 동등한 명령으로 변경 범위 최종 확인
3. plan.md의 "비범위" 침범 여부 확인
4. 종합 보고서 작성:
* 완료한 변경 사항
* 통과한 검증
* 영향도 — 이번 변경이 영향을 미치는 다른 기능/API/이벤트 (PR의 "영향도 및 주의사항" 섹션에 사용)
* 발견했지만 손대지 않은 사항
* 추후 처리 권장 사항
