# /system-design — 전체 시스템 설계 (엔티티 뼈대 + 서비스 경계 + 핵심 API 계약)

프로젝트 시작 시점(또는 새로운 도메인 영역 추가 시) 1회 실행한다. 요구사항 개요를 입력받아 시스템 전체의 핵심 도메인 엔티티 관계(ERD 뼈대), 서비스/모듈 경계, 경계를 넘는 핵심 API 계약의 큰 그림을 설계 문서로 작성한다.

이 문서는 "뼈대"다 — 모든 필드와 엔드포인트를 확정하지 않는다. 개별 기능 구현 시 `/spec-design`이 이 뼈대를 기반으로 세부 필드/엔드포인트를 채운다.

**사용법**
- `/system-design {요구사항 개요 경로}` — 해당 문서를 읽어 설계한다
- `/system-design` — 대화 중 붙여넣은 요구사항 개요로 설계한다

---

## Step 1. 입력 받기

파일 경로가 주어지면 해당 파일을 읽는다. 그렇지 않으면 사용자가 붙여넣은 내용을 요구사항 개요로 사용한다.
요구사항 개요가 없으면 사용자에게 요청한다.

`docs/system-design.md`가 이미 존재하면 함께 읽는다 — 이번 설계는 "신규 작성"이 아니라 "기존 뼈대에 새 도메인 추가"일 수 있다.
`docs/arch-snapshot.md`가 존재하면 함께 읽는다 — 기존 시스템의 서비스 구성(serverA/B/C 등)을 파악한다.

---

## Step 2. 핵심 도메인/엔티티 식별

요구사항 개요에서 핵심 도메인 엔티티를 식별한다. 이름과 한 줄 책임만 — 필드는 다루지 않는다.

`docs/system-design.md`가 있다면, 이미 정의된 엔티티는 다시 나열하지 않고 신규/변경되는 것만 식별한다.

```markdown
| 엔티티 | 책임 (한 줄) |
|---|---|
| {엔티티명} | {이 엔티티가 표현하는 것} |
```

---

## Step 3. 서비스/모듈 경계 정의

각 엔티티가 어느 서비스/모듈에 속하는지 정의한다.

- 기존 시스템이라면 `docs/arch-snapshot.md`의 서비스 구성(serverA/B/C 등)과의 관계를 우선 명시한다.
- 신규 서비스/모듈이 필요하면 이유를 함께 적는다.

```markdown
| 서비스/모듈 | 담당 엔티티 | 비고 |
|---|---|---|
| {서비스명} | {엔티티 목록} | {신규/기존, 신규라면 이유} |
```

---

## Step 4. ERD 뼈대

엔티티 간 관계만 표현한다. 필드는 관계 식별에 필요한 최소한(PK/FK 등)만 적는다 — 전체 필드 목록은 기능 구현 시 `/spec-design`에서 채워진다.

```markdown
### {엔티티명}
- 핵심 식별/관계 필드: {PK, FK 등 — 최소한만}
- 관계: {다른 엔티티와의 관계}
```

---

## Step 5. 핵심 API 계약 초안 (경계를 넘는 것만)

서비스 간, 또는 프론트엔드-백엔드 간 경계를 넘는 API/이벤트만 다룬다. 한 서비스 내부에서만 쓰이는 API는 다루지 않는다 — 이런 것은 기능 구현 시 `/spec-design`에서 정의된다.

엔드포인트 이름과 한 줄 설명, 호출 방향만 — 요청/응답 필드 전체 테이블은 다루지 않는다.

```markdown
### {METHOD} {경로} (또는 {Kafka 토픽명})
- 방향: {호출 서비스} → {수신 서비스}
- 설명: {한 줄}
```

---

## Step 6. 설계 문서 생성 및 저장

Step 2~5의 결과를 합친다.

```markdown
# 시스템 설계: {프로젝트/도메인명}

- 작성일: YYYY-MM-DD
- 관련 요구사항 개요: {경로 또는 "대화 중 입력"}

## 핵심 엔티티
{Step 2 결과}

## 서비스/모듈 경계
{Step 3 결과}

## ERD 뼈대
{Step 4 결과}

## 핵심 API 계약 (경계 간)
{Step 5 결과}

가정: {불명확해서 임의로 채운 부분이 있다면 명시}
```

- `docs/system-design.md`가 없으면 새로 생성을 제안한다.
- `docs/system-design.md`가 이미 있으면, 기존 내용을 덮어쓰지 않고 새 섹션으로 추가할 것을 제안한다 (`## 핵심 엔티티 (추가: {도메인명})` 형식으로 기존 섹션 뒤에 append).

저장할까요? (y/n)

저장 후, `/spec-draft` 또는 `/spec-design`으로 개별 기능 설계를 이어갈 수 있음을 안내한다 — `/spec-design`은 `docs/system-design.md`가 있으면 자동으로 참고해 정의된 엔티티/서비스 경계와의 일치 여부를 확인한다.
