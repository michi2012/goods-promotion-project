# /spec-draft — 요구사항 명세서 다듬기

부실한 요구사항 초안을 받아 User Story + Given-When-Then 형식의 정식 명세서로 재작성한다.
`/spec-to-tickets`에 바로 입력할 수 있는 형태로 만드는 것이 목표다.

**사용법**
- `/spec-draft {파일 경로}` — 해당 파일을 읽어 다듬는다
- `/spec-draft` — 대화 중에 붙여넣은 초안 내용을 다듬는다

---

## Step 1. 초안 입력 받기

파일 경로가 주어지면 해당 파일을 읽는다. 그렇지 않으면 사용자가 직접 붙여넣은 내용을 초안으로 사용한다.
초안이 없으면 사용자에게 요청한다.

입력이 Linear 이슈 식별자(`/project-plan`이 생성한 기능 이슈 등, 예: MIC-10)이면 `get_issue`로 해당 이슈를 조회해 `title`/`description`을 초안으로 사용한다.

---

## Step 2. 빠진 정보 파악

아래 항목을 기준으로 초안을 점검한다. **초안에 이미 있는 내용은 건너뛰고, 빠진 것만 질문한다.**

```
점검 항목:
1. 배경 — 이 기능이 왜 필요한가 (어떤 문제를 해결하는가)
2. 사용자 역할 — 누가 사용하는가 (한 명일 수도, 여러 역할일 수도)
3. 핵심 시나리오 — 정상적으로 동작할 때의 흐름
4. 예외/에러 케이스 — 실패하거나 비정상적인 입력이 들어왔을 때
5. 완료 기준 — 이 기능이 "다 됐다"고 판단할 수 있는 조건
6. 비범위 — 이번에 의도적으로 다루지 않는 것 (있다면)
7. 기술적 고려사항 (있다면) — 성능/확장성/예외 케이스 처리 관점에서 미리 고민해둔 게 있는가
```

수집이 끝나면 Step 3으로 진행한다.

---

## Step 3. 정식 명세서 생성

User Story와 Given-When-Then 형식으로 재작성한다. 초안에 없는 내용을 임의로 지어내지 않는다 — 추측이 필요한 부분은 "가정:"으로 명시한다.

```markdown
# 명세서: {제목}

## 배경
{이 기능이 왜 필요한지 — 해결하려는 문제}

## User Story
- As a {사용자 역할}, I want {원하는 기능}, so that {얻고자 하는 가치}

## 시나리오 (Given-When-Then)

### 시나리오 1: {정상 흐름 제목}
- Given {전제 조건}
- When {사용자 행동}
- Then {기대 결과}

### 시나리오 2: {예외 케이스 제목}
- Given {전제 조건}
- When {예외 상황 발생}
- Then {기대 결과 — 에러 처리 방식 포함}

## 수용 기준 (Acceptance Criteria)
- [ ] {검증 가능한 기준 1}
- [ ] {검증 가능한 기준 2}
- [ ] {검증 가능한 기준 3}

## 비범위 (Out of Scope)
- {이번에 다루지 않는 것}

## 기술적 고려사항 (Technical Considerations) — 있는 경우에만 포함
- {성능/확장성/예외 케이스 처리 관점에서 사용자가 언급한 고려사항}

가정: {불명확해서 임의로 채운 부분이 있다면 명시}
```

---

## Step 4. 저장

생성된 명세서를 저장할 경로를 제안한다.

```
docs/specs/{kebab-case-제목}.md
```

저장할까요? (y/n)

사용자가 승인하면 해당 경로에 파일을 생성한다. `docs/specs/` 디렉토리가 없으면 함께 생성한다.
저장 후, 엔티티/API 계약 설계가 필요하면 `/spec-design {저장된 경로}`을 먼저 실행할 수 있고, 그렇지 않으면 바로 `/spec-to-tickets {저장된 경로}`로 티켓을 생성할 수 있음을 안내한다.
